## 1. 摘要

本 PRD 定义「免 Key 兜底翻译源」：内置 Google 与微软翻译的免 Key 公共端点作为 provider，使用户在未配置任何 LLM 或传统 API Key 时也能开箱即用、直接划词翻译。这是 v0.1「不内置任何模型接口」原则在 v0.2 的边界调整——为降低首次使用门槛，内置传统翻译的公共端点作为兜底。

## 3. 背景

v0.1 明确「不内置任何模型接口」，用户必须自行配置 OpenAI 兼容接口或本地 Ollama 才能翻译。这对普通阅读用户门槛过高——许多人没有 API Key，也不知道如何本地部署模型，导致插件装上即不可用。

v0.2 决定调整这条原则：**内置传统翻译接口的免 Key 公共端点作为兜底**，让用户开箱即用。选择传统翻译接口（而非内置某个 LLM）作为兜底的原因：

- 传统翻译接口（Google / 微软）有长期稳定的非官方公共端点，无需 Key 即可调用，适合兜底。
- LLM 接口几乎都需要 Key，无法做到真正免 Key 开箱即用。
- 兜底只求「能用」，传统机翻质量足够应急；追求质量时用户仍可配置 LLM。

本 PRD 依赖功能事项 1（unified-adapter）提供的 `TranslationProvider` 接口与注册表，在其上实现 google / microsoft 两个传统 provider 并注册为兜底源。

**关键现实与风险**：所谓「免 Key 公共端点」多为**非官方端点**（如 Google 翻译网页端点、Edge 浏览器翻译端点），稳定性有限：可能被限流、封禁、或在部分地区不可达（Google 翻译在中国大陆不可达）。本 PRD 以「多源兜底 + 明确错误提示 + 人工切换」应对，不做自动降级。

## 4. 目标

**目标**：用户未配置任何翻译源时，划词翻译可直接返回译文（走免 Key 兜底源），实现真正的开箱即用。

**对齐愿景**：产品策略定位「让任何人都能用上最好的大模型翻译」——但「任何人」的前提是先能「用上」。兜底源降低了零门槛体验，让普通用户先用起来，再引导他们配置 LLM 获得更好质量。

**关键结果（SMART）**：

| 编号 | 关键结果 | 衡量方式 |
|---|---|---|
| KR1 | 至少 2 个免 Key 传统 provider（google、microsoft）实现并注册到适配层 | 代码审查 |
| KR2 | 用户未配置任何源时，划词翻译返回译文，无需任何配置 | e2e：全新状态划词成功 |
| KR3 | 兜底源失败时返回明确错误提示，引导用户切换或配置其它源 | 单元测试覆盖错误路径 |
| KR4 | 隐私声明已更新，明示默认外传文本到 Google/微软 | 隐私声明审查 |

## 5. 细分市场

- **普通阅读用户**（待完成任务：阅读外文页面，没有 API Key，只想马上看懂）：本 PRD 的核心受益者。零配置即可翻译，消除首次使用门槛。
- **技术用户 / 开发者**：兜底源作为「应急方案」，他们仍可配置自有 LLM/Key 追求质量与隐私。本 PRD 不影响其既有路径。

**约束**：
- 兜底源为非官方公共端点，稳定性与可达性无保证。
- 中国大陆环境下 Google 翻译不可达，需 microsoft（或其它可达源）兜底。
- 必须在隐私层面如实告知：默认会将待翻译文本外传给 Google/微软。

## 6. 价值主张

| 客户任务 / 需求 | 客户获得 | 客户避免的痛点 | 我们做得更好的地方 |
|---|---|---|---|
| 装上插件就想用 | 零配置开箱即用，划词即译 | 被迫先研究 API Key / 本地模型 | 多数翻译插件需配 Key；兜底让首次体验顺畅 |
| 应急看懂外文 | 传统机翻质量足够应急 | 无可用源时空白 | 内置多源，单点不可达仍有备选 |
| 知道数据去哪 | 隐私声明明示外传到哪 | 不知文本被发给谁 | 如实告知并提供配置自有源以保护隐私的路径 |

## 7. 解决方案

### 7.1 UX / 原型

用户侧无新增独立界面。兜底体现为「行为」：用户未配置任何源时，划词翻译自动走兜底源，浮层正常展示译文。配置页（功能事项 4）会展示当前生效源（含兜底源）与是否为兜底状态。

错误反馈：当兜底源不可达时，划词浮层展示明确提示（如「兜底翻译源不可达（可能受限流或地域限制），请在配置页添加自有翻译源」），由统一错误模型（功能事项 1）产出。

> 详细 UX 设计（差异性交互、状态、反馈、可访问性、视觉原型）待 issue 创建后由 prodflow-prd revise 模式补全（参考 prodflow-ux-evaluate 技能产出）。

### 7.2 核心功能

**7.2.1 免 Key 传统 Provider 实现**

在功能事项 1 的 `TranslationProvider` 接口上实现两个 provider：

- **google**：调用 Google 翻译免 Key 公共端点（非官方网页端点），解析返回结构提取译文。
- **microsoft**：调用微软翻译免 Key 端点（如 Edge 浏览器翻译端点等可达方案），解析提取译文。

两个 provider 均**无需用户填写 Key**，端点内置。实现细节（确切 URL、请求参数、响应解析）由研发在实现时确定并验证可达性，本 PRD 只约束行为契约：输入 `TranslateRequest`，输出 `TranslateResult`，失败归入统一错误模型。

**7.2.2 兜底启用逻辑**

- 适配层路由（功能事项 1）解析「当前生效源」时：若用户未配置任何源、且未显式启用其它源，则自动使用兜底源。
- 兜底源优先级：google 优先，不可达时由用户人工切换到 microsoft（本轮不做自动降级）。
- 一旦用户配置并启用了任意自有源（LLM 或传统 API Key），兜底源不再生效（被用户源覆盖）。

**7.2.3 错误处理与提示**

兜底源失败时按统一错误模型归类：
- 网络错误 / 源不可达（含地域不可达）：提示「兜底翻译源不可达，请在配置页添加自有翻译源」。
- 限流：提示「兜底源繁忙，请稍后再试或配置自有源」。

**7.2.4 host_permissions 声明**

兜底源域名需在 manifest `host_permissions` 声明，确保 background 能跨域请求。由研发在 wxt.config.ts / manifest 配置中落实。

### 7.3 技术

- Provider 实现位于 `shared/`（与 LLM provider 同层），由 background 经适配层调用。
- 端点内置为常量，不暴露给用户编辑（兜底源不可编辑，只能启用/禁用或被自有源覆盖）。
- 响应解析需处理非官方端点的非标准 JSONP / 数组结构，做好容错。

### 7.4 假设

| 假设 | 验证方式 | 失败信号 |
|---|---|---|
| Google / 微软存在可用的免 Key 公共端点 | 研发实现时实测 | 两源均不可用，兜底不可行 |
| microsoft 端点在中国大陆可达 | 实测 | 大陆用户无兜底，需额外源 |
| 非官方端点响应结构可稳定解析 | 实测 + 容错 | 响应结构频繁变动致解析失败 |
| 用户接受默认外传文本到 Google/微软 | 隐私声明明示 | 用户反馈隐私顾虑 |

## UX 设计

### UX 依据

- Issue: `#2`（https://github.com/Aiden-FE/llm-translator/issues/2）
- 版本: v0.2
- 知识来源: [knowledges/ux/index.md](../../../../ux/index.md)（设计原则：轻量不打扰、隐私透明）、[knowledges/ux/interaction-patterns.md](../../../../ux/interaction-patterns.md)（划词浮层）、[knowledges/ux/accessibility.md](../../../../ux/accessibility.md)
- 设计假设: 用户接受「未配置时默认走免 Key 兜底（文本外传到 Google/微软）」的默认行为，只要在配置页如实告知即可——该假设待用户验证，若用户反馈默认外传不可接受，需改为「首次使用时显式确认」

### 视觉设计

> 本 PRD 无新增独立用户界面（兜底为运行时行为，配置页兜底状态展示由 [../4-source-picker-ui/PRD.md](../4-source-picker-ui/PRD.md) 承担）。沿用现有划词浮层视觉，不单独产出视觉原型。

### 交互逻辑（差异）

通用划词流程引用 [knowledges/ux/interaction-patterns.md](../../../../ux/interaction-patterns.md)，不重复展开。本 PRD 差异：

1. **兜底无感启用**：用户未配置任何源时，划词翻译自动走兜底源，**浮层不额外标注「兜底」字样**，遵循「轻量不打扰」原则——译文正常展示，体验与配置源一致。
2. **隐私透明落在配置页**：兜底的「文本默认外传到 Google/微软」告知放在配置页（当前生效源展示处），不在每次翻译的浮层中重复打扰。配置页提示文案约束见 [../4-source-picker-ui/PRD.md](../4-source-picker-ui/PRD.md)。
3. **兜底失败反馈**：沿用 [../1-unified-adapter/PRD.md](../1-unified-adapter/PRD.md) 统一错误模型，兜底源失败按「源不可达 / 限流」反馈，引导文案强化「请在配置页添加自有翻译源」。

### 业务约束

- 兜底源对用户不可编辑（端点内置），只能被自有源覆盖或在未来版本禁用。
- 兜底优先级：google 优先，不可达时由用户人工切换到 microsoft（本轮不做自动降级）。
- 隐私约束：兜底启用期间，待翻译文本必然外传给 Google/微软，配置页必须如实反映当前生效源为兜底源及外传去向。

### 状态变化（差异）

浮层状态沿用第 1 项四类错误子状态。兜底场景下高频出现的两类：

| 状态 | 反馈 |
|---|---|
| 成功（兜底） | 译文文本（与配置源成功态无差异） |
| 失败-源不可达（兜底） | ❌ 翻译源不可达 + 引导：请在配置页添加自有翻译源 |
| 失败-限流（兜底） | ❌ 翻译源繁忙 + 引导：请稍后再试或配置自有源 |

### 设备适配（差异）

无差异。浮层响应式沿用 [knowledges/ux/design-system.md](../../../../ux/design-system.md)。

### 可访问性（差异）

- 兜底状态对辅助技术无额外要求（浮层不标注兜底）。
- 隐私告知在配置页以文字呈现，不依赖颜色，需满足对比度 AA（约束第 4 项配置页实现）。

### 验收补充

- 未配置任何源时划词翻译返回译文，浮层无「兜底」额外标注（轻量）。
- 兜底源失败时返回「源不可达/限流」可读提示并引导配置自有源。
- 配置页如实展示兜底为当前生效源及外传去向（约束第 4 项）。
- 隐私声明已更新默认外传行为（见 [../../privacy/PRIVACY-POLICY.md](../../privacy/PRIVACY-POLICY.md)）。
