# PRD:视觉主题改造（2-visual-theme）

> 版本:v0.3 第 2 项 · 优先级 P0 · 关联 Issue 待创建 · 创建日期 2026-07-07 · 状态 draft

## 1. 摘要

本 PRD 定义 LLM Translator v0.3 视觉主题改造：在第 1 项 ui-rewrite 新设计系统基座上，弃用现有灰白黑冷色板，定义并应用适度明亮鲜艳（不抢眼）的主题，使全应用视觉焕新、提升商店首发吸引力，同时保持阅读舒适与可访问性达标。设计 token 落档 design-system.md，作为后续开发的统一基准。

| 角色 | 负责人 | 备注 |
|---|---|---|
| 产品 | Prodflow | 需求与验收 |
| 前端 | 待分配 | tailwindcss theme config + CSS 变量主题定义与应用 |
| UX 设计 | web-design-engineer | 色板 / 设计 token 规格（待产出，不硬编色值） |

## 3. 背景

- **当前状态**：design-system.md 色彩为灰白黑冷色调——浮层背景 `#1F2937`、文字 `#F9FAFB`、边框 `#E5E7EB`/`#D1D5DB`、错误 `#DC2626`、成功 `#16A34A`。整体偏沉闷，在商店缩略图中缺乏辨识度。
- **为什么现在做**：2026-07-06 战略转向增长优先——中小翻译插件竞品密集，产品美观是抢占市场的关键抓手；v0.3 上线前须全面美化，弃灰白黑冷色改适度明亮鲜艳主题，以视觉吸引力突围。v0.3 同时是商店首发版本，第一眼视觉直接决定安装转化。
- **变化点**：v0.3 第 1 项 ui-rewrite 引入 tailwindcss + shadcn/ui 重构全套组件，提供新设计系统基座；本项在该基座上集中定义色彩/字体/间距/圆角/阴影 token，替换现有冷色板，引入有彩色作主色与强调色。具体色板待 UX 规格（web-design-engineer）产出。

## 4. 目标

**目标**：弃用纯灰白黑冷色，定义并应用适度明亮鲜艳的主题；产出设计 token 落档 design-system.md；全应用视觉焕新、吸引力提升但保持阅读舒适与可访问性达标。

**为什么重要**：商店首发是增长优先阶段的关键触点，用户在商店浏览时 3 秒内决定是否停留；冷色调沉闷难以在竞品密集的商店突围，明亮鲜艳主题直接提升安装转化。已安装用户的划词浮层叠在网页上，主题须「鲜艳但不抢眼」以保证阅读舒适。

**战略对齐**：直接对应 strategy「产品美观是增长抓手——弃用灰白黑冷色调、改适度明亮鲜艳（不抢眼）的主题，在中小竞品密集的市场以视觉吸引力突围」与 roadmap v0.3 应用美化要求。

**关键结果（SMART OKR）**：
- O：视觉主题焕新，提升商店首发吸引力
  - KR1：冷色板（`#1F2937`/`#F9FAFB`/`#E5E7EB`/`#D1D5DB` 等）100% 替换为适度明亮鲜艳主题，全应用无残留冷色硬编码
  - KR2：设计 token（主色、强调色、语义色 success/error/warning/info、字体、间距、圆角、阴影）落档 design-system.md
  - KR3：新主题可访问性对比度达标（accessibility.md WCAG AA，正文文字与背景对比度 ≥ 4.5:1）

## 5. 细分市场

按待完成任务定义用户：

- **商店浏览用户（首次接触产品）**：任务是在 Chrome/Firefox/Edge 商店浏览翻译插件时被吸引并安装。当前冷色调截图在商店缩略图中缺乏辨识度，难以从同类产品中脱颖而出。约束：商店截图是第一触点，用户 3 秒内决定是否停留；主题须在缩略图尺寸下仍有吸引力。
- **已安装使用用户**：任务是在划词翻译、配置翻译源时获得舒适阅读体验。约束：翻译浮层叠在原网页内容之上，主题色不可抢眼到干扰原网页阅读；长时间使用不产生视觉疲劳。

## 6. 价值主张

| 客户任务 | 现状痛点 | 升级后收益 |
|---|---|---|
| 商店浏览被吸引 | 冷色调沉闷，商店缩略图缺乏辨识度 | 明亮鲜艳主题提升第一眼吸引力，更多安装转化 |
| 划词阅读译文 | 冷灰浮层略显压抑 | 适度鲜艳但舒适的主题，阅读愉悦不疲劳 |
| 配置翻译源 | 冷色 UI 缺少操作引导感 | 明确色彩语义（主色引导操作、语义色反馈状态），操作更直观 |

**比竞品优在哪**：多数翻译插件要么纯白单调、要么高饱和刺眼；本主题取「适度明亮鲜艳、不抢眼」的中间路线，兼顾吸引力与阅读舒适，并保证可访问性对比度达标（WCAG AA），在商店审核与用户体验上双优。

## 7. 解决方案

### 7.1 UX / 原型（高层流程）

用户感知流程：打开扩展（popup / options 配置页）或划词触发翻译浮层 → 全应用呈现统一明亮鲜艳新主题（主色引导操作、语义色反馈状态、有彩色替代冷灰）→ 视觉焕新但阅读舒适、不干扰原网页内容。

具体色板、视觉原型、状态色映射待 UX 规格（web-design-engineer）产出，详见末尾「## UX 设计」章节占位。

### 7.2 核心功能

1. **定义设计 token**：在第 1 项 ui-rewrite 的 tailwindcss/shadcn 基座上，集中定义色彩 token（主色、强调色、语义色 success/error/warning/info）、字体、间距、圆角、阴影，替换现有冷色板（`#1F2937`/`#F9FAFB`/`#E5E7EB`/`#D1D5DB` 等）。具体色值待 UX 规格（web-design-engineer）产出。
2. **全应用主题应用**：popup、options 配置页、翻译浮层、触发按钮全部切换到新主题 token；确保全应用无冷色硬编码残留（grep 验证）。
3. **CSS 变量驱动主题**：主题 token 以 CSS 变量集中定义，tailwindcss theme config 引用变量，便于未来扩展（如暗色模式、多主题切换）。
4. **设计 token 落档**：更新 design-system.md 的色彩/字体/间距/圆角/阴影表，记录新色板与 token 定义，替换现有冷色板记录，作为后续开发的统一基准。
5. **可访问性保障**：新主题正文文字与背景对比度满足 WCAG AA（accessibility.md，≥ 4.5:1）；语义色不作为唯一区分手段（同时保留文字/图标说明，沿用 accessibility.md 既有约束）。

### 7.3 技术

- **入口**：tailwindcss theme config（colors / spacing / radius / shadow）+ CSS 变量定义文件；与第 1 项 ui-rewrite 协同，在其 tailwindcss/shadcn 基座上定义 token，不重复搭建基座。
- **复用**：复用第 1 项 ui-rewrite 的 tailwindcss/shadcn 组件结构与基座；本项聚焦 token 定义与应用，不重构组件结构（属第 1 项范围）。
- **类型**：无新增类型；设计 token 为配置层，不涉及 `shared/types.ts` 变更。
- **存储**：不涉及存储变更（纯视觉层，无 chrome.storage 读写改动）。
- **兼容**：与第 1 项 ui-rewrite 协同推进，不破坏现有划词翻译/配置链路；v0.2 既有 e2e 用例回归保证不退化。具体色值不硬编进 PRD，标注「待 UX 规格（web-design-engineer）产出」。

### 7.4 假设

- 假设第 1 项 ui-rewrite 的 tailwindcss/shadcn 基座先于本项就绪，本项在其上定义 token（依赖 1→2 顺序执行）。
- 假设「适度明亮鲜艳、不抢眼」可通过有彩色主色 + 中性辅色实现，无需引入图片/渐变等重资源。
- 假设新主题在浮层叠网页场景下不干扰原网页阅读（浮层自带不透明背景，主题色限于浮层与扩展 UI 内部）。
- 假设 web-design-engineer 能在本项开发窗口内产出色板规格，不阻塞 token 定义与落地。
